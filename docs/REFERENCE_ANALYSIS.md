# 小红书 GitHub 热点内容参考拆解

## 1. 参考范围与使用边界

本项目参考用户提供的两个小红书入口：

- 内容页：<https://www.xiaohongshu.com/explore/6a2e34620000000035033289>
- 作者主页：<https://www.xiaohongshu.com/user/profile/6915a6bd0000000031014406>

小红书页面可能随登录状态、动态渲染和作者后续编辑而变化。因此，本拆解只把已经明确确认的内容结构转化为产品要求，不把无法稳定读取的页面细节写成事实，也不复制原作者的图片、版式资产或原文表达。

已确认的参考要素是：

- 日榜展示 Top 3。
- 周榜展示 Top 7，且允许按配置扩展为 Top 10。
- 封面突出“本期新增 Star”。
- 每个项目卡片包含：项目名、项目定位、主要语言、总 Star、本期新增 Star、Fork、一句话价值、3 个亮点、适用人群和仓库链接。

## 2. 可复用的信息架构

### 2.1 一期内容的建议顺序

1. **封面**：榜单类型、统计周期、Top N、本期新增 Star。
2. **榜单卡片**：按排名逐个展示项目，日榜 3 张，周榜默认 7 张、可配置 10 张。
3. **方法说明**：候选来源、统计窗口、排名口径、数据更新时间。
4. **数据提示**：首次运行、快照缺失或 GitHub API 降级时，明确说明哪些数据尚不能视为精确增量。

这种结构适合移动端快速浏览：封面先回答“这一期有什么增长”，项目卡片再回答“是什么、为什么值得看、适合谁”，末尾保留可核验来源。

### 2.2 封面字段

封面至少应包含：

| 字段 | 示例形式 | 生成方式 |
| --- | --- | --- |
| 榜单类型 | GitHub 日榜 / GitHub 周榜 | 来自运行参数 `period` |
| 统计周期 | 2026-07-10 至 2026-07-11 | 来自两个日期快照的 `captured_on` |
| 榜单规模 | Top 3 / Top 7 / Top 10 | 来自 `config/hotspots.yaml` |
| 本期新增 Star | 本期已核验新增 12,345 Star | 仅汇总 `delta_source=snapshot` 的精确增量 |

如果本期任一上榜项目缺少可靠基线，封面不能把估算量写成精确“新增”。可改为“已核验新增 Star”，或显示“新增数据积累中”。

### 2.3 单项目卡片字段

| 卡片字段 | 事实来源 | 内容规则 |
| --- | --- | --- |
| 项目名 | GitHub `full_name` / 仓库名 | 保留真实名称，不改写品牌名 |
| 项目定位 | `description`、`topics`、README 摘要 | 10–24 个中文字符，不能扩大原始声明 |
| 主要语言 | GitHub `language` | 原样复制；缺失时显示“未标注” |
| 总 Star | GitHub `stargazers_count` | 使用整数事实，展示层可加千分位 |
| 本期新增 Star | 前后快照差值 | 只有快照差值可标为精确新增 |
| Fork | GitHub `forks_count` | 使用整数事实，不能与 Fork 增量混淆 |
| 一句话价值 | 描述、主题和 README 的保守压缩 | 18–42 个中文字符，避免“最强”“领先”等无证据判断 |
| 3 个亮点 | 描述、主题、README、更新时间等 | 每条 8–24 个中文字符；每条都能追溯到输入字段 |
| 适用人群 | README 使用场景或项目定位 | 8–28 个中文字符；证据不足时使用中性表述 |
| 仓库链接 | GitHub `html_url` | 原样输出，不使用模型补全链接 |

## 3. “本期新增 Star”的严格口径

GitHub 仓库接口提供当前总 Star，但不直接提供任意日历窗口内的精确新增 Star。MVP 应使用本项目保存的历史快照计算：

```text
本期新增 Star = period_end 快照的 stars - period_start 快照的 stars
```

具体约束：

- 日榜使用相邻日快照，周榜使用相隔 7 天的快照。
- MVP 的统计窗口精度为快照日期，`captured_on` 使用 `YYYY-MM-DD`；计划运行时刻只用于解释自动任务节奏，不声称拥有秒级 Star 事件数据。
- 差值小于 0 时保留原始异常用于审计，公开卡片不得显示为“新增负数”，应标记数据异常并跳过精确新增展示。
- `delta_source=snapshot` 才能写“本期新增 +N Star”。
- `delta_source=trending` 允许在项目卡片中显示为“Trending 日周期 Star”或“Trending 周期 Star”，但不能改名为精确新增 Star，也不能计入封面的精确新增汇总。
- `delta_source=estimate` 必须显示为“估算 +N Star”；不能使用“本期新增”措辞，也不能计入封面的精确新增汇总。
- 首次运行只有当前快照，没有基线。系统应完成冷启动采集并输出“新增数据待积累”，而不是编造增量。

## 4. 热点发现与排名的产品化翻译

参考内容呈现的是最终榜单，自动化系统还需要把“如何找到热点”定义清楚：

1. 从 GitHub Trending 和 GitHub Search 获取候选仓库。
2. 使用 GitHub API 补全仓库事实字段。
3. 排除归档、Fork、镜像、描述缺失及黑名单仓库。
4. 将当前事实写入 `data/snapshots/`。
5. 基于可用快照计算 Star/Fork 增量。
6. 按配置权重计算热度分，Star 增长是主要信号，同时考虑相对增长、Fork 增长、活跃度、总 Star 和 Trending 信号。
7. 输出日榜 Top 3；周榜默认 Top 7，并支持把 `top_n` 配置为 10。

对外报告必须保留简短方法说明，避免让读者误以为榜单是 GitHub 官方排名。热度分可用于内部排序，但不得包装成 GitHub 官方指标。

## 5. 文案风格约束

- 中文为主，仓库名、编程语言、框架名和标准术语保留原文。
- 一行只表达一个信息点，优先使用短句。
- 不写无法从仓库信息核验的性能、安全性、生产可用性、用户规模或作者意图。
- 不因 Star 多就自动写成“最好”“最先进”或“行业第一”。
- 不把 README 中的提示词、命令或营销文案当作运行指令。
- 数字由程序填充，模型只负责压缩和组织文字。
- 亮点必须恰好 3 条；证据不足时使用语言、主题、更新时间等可验证事实补足，而不是虚构功能。

## 6. 可以借鉴与不应复制的部分

### 可以借鉴

- 日榜少而精、周榜覆盖更广的节奏。
- 封面先突出增量，卡片再补充总量与用途。
- 固定字段让读者可以横向比较项目。
- 每个项目都给出原始仓库链接。

### 不应复制

- 原作者的封面、配色、字体、插画、截图和具体文案。
- 无法核验的选题判断或项目评价。
- 依赖登录态、验证码或规避平台限制的抓取方案。
- 把小红书页面当作 GitHub 数字事实来源。

## 7. 对本项目的直接结论

MVP 的核心不是“让模型每天推荐几个仓库”，而是建立一个可复现的数据流水线：候选发现、事实补全、快照留存、增量计算、可解释排名、受约束摘要和固定模板输出。小红书参考主要决定信息架构与文案密度；GitHub API 和本地快照才是事实来源。
