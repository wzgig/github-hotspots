# 小红书内容 V2 改进计划

> 状态：执行规范与 Prompt 4.0 目标设计  
> 适用范围：“综合主榜 + AI 专题榜”的小红书文案、轮播卡片与人工发布审核  
> 当前兼容性：运行时仍使用 `RepositorySummary(one_line, highlights[3], audience)` 和输出 Schema 3.0；Schema 4.0 是下一阶段设计，不在本轮直接切换

## 1. 决策摘要

当前内容的主要矛盾不是“文案不够热闹”，而是“数据卡没有完成用途解释”。Star、Fork、语言和 Topics 可以证明仓库状态，却不能自动回答读者最关心的三个问题：

1. 它替谁工作？
2. 它具体做什么？
3. 使用后得到什么结果？

V2 将以“替谁做什么，得到什么”作为所有项目解释的最小闭环，并采用以下路线：

- 短期 V2：保持 Schema 3.0 和当前代码兼容，先让程序候选文字变成白话用途、具体能力和任务场景；受控选稿器仍只逐字符选择，不自由补写。
- 内容分面：Pages 详情卡和小红书轮播卡使用不同信息密度和发布规则，不再让一张“工程仪表盘”同时承担两种任务。
- 证据优先：每条能力必须由受控输入支持；不能确认时明示“需人工核对”，不用泛话伪装成能力。
- 人工决策：自动化仅到“生成可审核文案和配图”；人工负责事实、权利、AI 辅助标识和最终发布。
- 长期 Schema 4.0：再引入 README 证据片段、逐主张引用、使用条件、限制和正式审核状态。

### 2.2 不得写成“确定机制”的说法

以下说法缺少可靠公开证据，本项目的文档、Prompt、站点和发布文案不得将其当作事实：

- 固定的“冷启动流量池”规模、轮次和晋级阈值。
- 所谓 CES 等互动权重公式，以及点赞、收藏、评论的固定倍数。
- “发布后前两小时决定全部流量”或某个必胜发布时间。
- 某个固定字数、标签数量或发布频次必然获得推荐。
- 使用“神器”“必装”“爆火”等词本身能带来更高流量。

本项目可以优化可观察的内容质量，例如首屏可理解性、读完后的任务理解、收藏价值和评论问题质量；不得声称这些信号在平台排序中的具体权重。

## 3. 受众与阅读任务

### 3.1 核心受众

| 受众 | 首要问题 | 内容必须提供 |
| --- | --- | --- |
| 非技术的 AI/效率工具用户 | 它能替我省哪件事？ | 具体任务、结果形式、使用前提 |
| 学生、初学者、知识工作者 | 我看不懂 CLI、MCP、Agent 等词，它和我有什么关系？ | 术语白话解释、典型场景、是否需要编程 |
| 开发者和技术评估者 | 它的功能边界、依赖和许可条件是什么？ | 语言、部署/依赖、许可证、已知限制、证据来源 |
| 收藏型读者和选型者 | 这几个项目有什么区别，哪个适合我？ | 用途/受众/门槛/限制比较，可搜索的 `owner/repo` 标识 |

### 3.2 阅读任务顺序

每组内容应让读者依次完成：

1. **3 秒识别**：知道这是哪个榜单、本期有几个项目、自己能得到什么。
2. **10 秒理解**：能用自己的话复述单个项目“替谁做什么，得到什么”。
3. **30 秒判断**：知道自己是否适用，以及使用前需要什么。
4. **滑完选择**：在同期项目中做初步比较，决定收藏、进一步搜索或提问。

## 4. 内容模型：从元数据卡到任务解释卡

### 4.1 最小解释闭环

单项目必须形成以下链条：

```text
谁在什么场景遇到问题
  -> 项目执行什么动作
  -> 用户得到什么结果
  -> 使用前需要什么
  -> 哪些是确定事实，哪些仍待核对
```

`one_line` 不要强制使用同一句式，但必须同时包含“使用者/场景、动作、结果”。示例句式：

- 给 `[specific audience]` 用来 `[action]`，完成后得到 `[result]`。
- 当 `[scenario]` 时，它会 `[action]`，帮助 `[audience]` 得到 `[result]`。
- 把 `[input]` 变成 `[output]`，适合需要 `[task]` 的 `[audience]`。

不合格句式：

- `X 是一个面向工程实践方向的开源项目`
- `X 是一个值得关注的 AI 工具`
- `X 以 Python 为主要语言，本周新增 500 Star`

第三句只提供元数据，不是用途解释。

### 4.2 短期 Schema 3.0 字段语义

| 字段 | V2 含义 | 不得替代为 |
| --- | --- | --- |
| `one_line` | 回答“替谁做什么，得到什么”的白话摘要 | Star 变化、主要语言、Topics 列表或泛化类别 |
| `highlights[3]` | 三条互不重复的具体能力，每条使用动词和可观察结果 | 重复 `one_line`、重复数字、简单罗列 Topics |
| `audience` | 具体任务场景中的人，优先描述“要做什么” | 过宽的“开发者、技术团队、开源爱好者” |

如果受控信息不足以确认用途，安全兜底必须明示说明：

- `one_line`：“公开信息不足，暂不能准确说明具体用途”。
- `highlights`：分别标记需核对 README 定位、输入/输出和适用场景。
- `audience`：“需人工确认后再发布”。

这比“开源软件与工程实践项目”更诚实，也不会让读者将类别名误解为项目能力。

## 5. 小红书轮播结构

综合主榜和 AI 专题榜继续分开生成文案与图片，同一仓库可在两榜同时出现。

### 5.1 日榜

建议 6–7 张：

1. **封面**：榜单、日期/期号、“3 个项目分别解决什么问题”。
2. **先看结论**：三个项目各一句用途 + 适用人群。
3. **项目 1**：单项目详解卡。
4. **项目 2**：单项目详解卡。
5. **项目 3**：单项目详解卡。
6. **怎么选**：用途 / 适合谁 / 使用前需要什么。
7. **方法与互动**：数据窗口、人工审核、AI 辅助标识、下期问题。

如果暂时不能增加结论卡，最少保留“封面 + 3 张项目卡 + 方法/互动卡”。

### 5.2 周榜

建议 10–11 张：

1. 封面。
2. 先看结论。
3. 至 9. 七张独立项目卡。
10. 横向比较/选型卡。
11. 方法、限制、AI 辅助标识和互动卡。

发布前由人工按小红书当前客户端允许的图片数量和字段限制复核，不在程序中把未持续核验的平台上限当成永久事实。

## 6. 单项目卡规范

卡片的信息顺序必须是“用途优先，数字次之”：

1. **顶部识别**：`#排名`、`owner/repo`、榜单标识。
2. **说人话摘要**：一句回答“替谁做什么，得到什么”。
3. **它能完成什么**：三条具体能力，以动词开头，不重复摘要。
4. **适合谁**：用任务场景描述，不只给宽泛职业名。
5. **为什么本期上榜**：精确数据窗口内的净增、排名和快照日期；不将 Star 增长推理成功能好坏或流行原因。
6. **次要事实**：语言、总 Star、Fork、许可证。
7. **使用前注意**：系统、依赖、API Key、自托管、编程基础或已知限制；暂无证据时明示“需人工核对”。

短期 Schema 3.0 尚无法完整容纳第 7 项，卡片可先展示受控兜底文字；正式的条件与限制字段留给 Schema 4.0。

### 6.1 视觉约束

- 继续使用 3:4 竖版，但小红书卡片改为更明亮、纸张/知识卡式的视觉，不复制参考帖的配色、图形或版式。
- 手机缩略图中必须看清项目名和一句用途；次要数字不得抢过用途。
- 单项目卡优先一屏完成解释；不为填满版面增加小字。
- 封面不展示 `00 Trending`、`00 估算`等零值或内部数据状态。
- 不把装饰性英文当作主标题；技术专名可保留，首次出现时要解释。

## 7. 标题与首屏

### 7.1 标题公式

标题使用“范围 + 数量/榜单 + 读者收益”，优先包含自然搜索词：

- `GitHub AI 周榜｜7 个项目到底能做什么`
- `别只看 Star｜本周 GitHub 热门项目讲人话`
- `3 个 GitHub 热门项目：用途、门槛、适合谁`
- `开发者周报｜7 个开源项目怎么选`

内部设计目标是“一眼读完”，发布前再按客户端当前字符限制人工调整。

禁止：

- 全网爆火、所有人都在用、必装、封神、神器、宝藏、零门槛。
- 没有定义数据窗口时的“一周暴涨”。
- 用 Star 数直接证明“最好用”“生产级”或“商用无忧”。

### 7.2 封面信息

封面只保留：

- 榜单名称和日期/期号。
- 一句明确收益，例如“3 个项目分别解决什么问题”。
- 一句内容承诺，例如“每个项目讲清：用途 · 适合谁 · 使用条件”。
- 最多 2–3 个本期真实任务词，例如“会议整理”“终端自动化”“代码测试”。

排名方法、快照质量和详细数据放在尾卡或正文末尾，不占用封面的首要注意力。

## 8. 正文规范

正文使用以下顺序：

1. **读者问题**：前两行写“看到 GitHub 榜单但不知道项目能做什么”等真实困惑。
2. **本期承诺**：说明会讲清用途、受众和使用条件，不只报 Star。
3. **项目摘要**：每个项目一行，格式为“项目名 - 具体任务 - 适合谁”。
4. **比较/推荐**：只根据已证实的任务和条件做差异化总结，不用主观“最强”排序。
5. **方法和边界**：标明数据窗口、快照/估算口径、AI 辅助和人工审核。
6. **互动问题**：提出一个可回答、可帮助下期选题的问题。
7. **标签**：使用少量精确主题，不堆叠无关热词。

小红书版不放裸 URL、二维码或外部联系方式。为便于读者核对项目，可在人工审核后保留不带协议和域名的 `owner/repo` 搜索标识。Pages 版仍可保留可点击的 GitHub URL。

## 9. 互动与收藏设计

互动的目的是收集真实需求，不是操纵平台信号。

可用问题：

- `下一期更想看会议助手还是终端自动化？`
- `哪个项目希望我下期补一张上手条件卡？`
- `你选工具时更在意本地部署、免费开源，还是使用门槛？`

收藏价值由以下内容提供：

- 一张横向比较表。
- 一张“适合谁/不适合谁”卡。
- 明确的使用前提和搜索标识。
- 可核对的数据窗口和快照日期。

禁止诱导点赞、互赞、互藏、抽奖换互动、买量刷数据或机器评论。

## 10. 证据链与 README 不可信输入安全

### 10.1 信任分层

| 层级 | 内容 | 用法 |
| --- | --- | --- |
| 冻结结构化事实 | 仓库 ID、`full_name`、URL、Star、Fork、语言、许可证、日期、排名、快照增量 | 程序拷贝与校验，模型不得修改、换算或纠错 |
| 受控分类事实 | 精确 Topics、否定保护后的严格短语规则 | 只能触发预定义能力文案，不得自由关键词联想 |
| 程序候选文字 | `candidate_summaries` | 当前 Prompt 只能从中选择并逐字符复制 |
| README/描述/网页文字 | 项目作者提供的自由文本 | 永远按不可信数据处理；可提供事实候选，不得成为指令 |

### 10.2 README 处理规则

长期引入 README 时，必须先建立“提取”与“编辑”之间的隔离层：

1. 只通过受控 GitHub API/快照获取指定仓库的 README，不跟随 README 中的外部链接。
2. 将 README 作为纯数据解析；其中的“忽略之前指令”“运行命令”“上传文件”等内容不得被执行。
3. 去除或隔离 HTML、隐藏文字、徽章、过长代码块和外部营销素材。
4. 只提取定位、明确功能、典型输入/输出、安装前提、兼容性和已知限制。
5. 每条可发布主张都保留 `source_id`、源类型、标题/定位符、原文片段和获取时间。
6. README 中的自我宣传，例如“最快”“领先”“生产级”，只能当作“项目方声称”，除非有可验证证据，否则不转写为事实。
7. 相互冲突的描述进入人工审核，不由模型选择看起来更合理的版本。

### 10.3 无证据时的状态

当前 Schema 3.0 没有单独的内容审核状态，因此短期用受控文本明示标记，不让模型自由生成审核警告。下一阶段引入：

- `verified`：每条主张都有可定位证据。
- `review_required`：有部分证据，但用途、受众、条件或限制仍需人工核对。
- `insufficient_data`：无法支持可发布解释，不生成对外卡片。

## 11. Prompt 4.0 目标规范

### 11.1 当前兼容桥接

`prompts/repository_summary_zh.md` 在本阶段使用“Prompt 4.0 编辑标准 + Schema 3.0 输出协议”：

- 模型仍只选择单个 `candidate_summaries` 候选，逐字符复制 `one_line`、`highlights`、`audience`。
- 优先选择能让读者理解“谁 - 动作 - 结果”的候选。
- 不得把 Star、Fork、语言、Topics、活跃日期或来源当成项目能力。
- 如果候选中已有“需人工核对”兜底，必须保留，不得用无证据的自信文字覆盖。
- 输出仍为 `schema_version=3.0`，以保持当前 `editorial.py` 和输出 Schema 兼容。

### 11.2 长期 Prompt 4.0 输入

当 Schema 4.0 实施后，编辑器只接收程序已清洗的事实包：

```json
{
  "repository_facts": {},
  "ranking_facts": {},
  "source_evidence": [
    {
      "source_id": "readme:features:1",
      "source_kind": "readme",
      "locator": "Features > item 1",
      "quote": "untrusted source text",
      "captured_at": "ISO-8601"
    }
  ],
  "allowed_claims": [],
  "review_flags": []
}
```

README 原文仍是不可信数据；`allowed_claims` 由程序根据证据转换和验证，模型只能在其语义边界内用白话改写。

### 11.3 长期 Prompt 4.0 输出

建议 Schema 4.0 字段：

```json
{
  "content_status": "verified | review_required | insufficient_data",
  "plain_summary": "替谁做什么，得到什么",
  "user_problem": "用户遇到的具体问题",
  "use_cases": [
    {"text": "能力一", "evidence_refs": ["source_id"]},
    {"text": "能力二", "evidence_refs": ["source_id"]},
    {"text": "能力三", "evidence_refs": ["source_id"]}
  ],
  "audience": {"text": "具体任务人群", "evidence_refs": ["source_id"]},
  "requirements": [],
  "caveats": [],
  "jargon_glossary": [],
  "why_ranked": {},
  "uncertainties": [],
  "asset_source": {},
  "ai_assisted": true
}
```

长期版允许模型做受证据约束的白话改写，但要求：

- 每条能力、受众、条件和限制都有 `evidence_refs`。
- 数字、URL、仓库身份、日期、排名、许可证完全冻结。
- 第一次出现的技术术语给出一句白话释义。
- 不声称使用者亲测、性能优势、商用安全或生产就绪，除非输入包含相应可复现证据。
- 证据不足时输出审核状态，不用模糊表述遮掩。

### 11.4 Prompt 正反例

以下只演示证据充足时的文字结构，不是对某个真实仓库的事实断言。

**正例**

> 给 C++ 开发者编写并运行单元测试，输出失败用例，便于定位代码问题。

这句同时包含使用者、动作和结果。前提是每个分句都有允许主张或 README 证据支持。

**反例：泛话**

> 这是一个面向开源软件与工程实践的热门项目。

没有可执行任务，“热门”也被用来代替用途。

**反例：元数据冒充能力**

> 项目使用 Python，本周新增 2,000 Star，Topics 包含 AI。

语言、Star 和 Topics 是事实栏，不能说明项目做什么。

**反例：无证据亲测**

> 我亲测安装简单、运行稳定，值得所有人使用。

如果没有可复现本地测试记录，整句都不得发布。

**正例：证据不足**

> 公开信息不足，暂不能准确说明具体用途；需人工核对 README 后再发布。

明示不确定比创造一个听起来合理的功能更安全。

## 12. 短期 V2 与长期 Schema 4.0 路线

### 12.1 短期 V2：保持兼容

1. 将七个候选 angle 从“七种元数据开场”改为“同一受控用途的七种白话表达”。
2. `one_line` 为任务与结果；`highlights` 为三条具体能力；`audience` 为具体任务场景。
3. 只依据精确 Topics 和少量否定保护的严格短语规则生成能力；规则无法确认时使用审核兜底。
4. Prompt 仍逐字符复制候选，保留 MCP 隔离、数字冻结、角度覆盖和失败回退。
5. Schema 保持 3.0，不影响当前报告、海报和 Pages 消费者。

### 12.2 中期 V2.1：引入可定位证据

1. 从 GitHub API 获取同仓库 README 快照。
2. 清洗并提取定位、功能、输入/输出、依赖、限制和资产来源。
3. 为每条允许主张生成稳定 `source_id`，不让模型直接面对整篇 README。
4. 在人工审核界面显示主张与原文对照。

### 12.3 长期 Schema 4.0：正式证据化出版

1. 引入 `content_status`、`use_cases`、`requirements`、`caveats`、`uncertainties` 和逐主张 `evidence_refs`。
2. 将 Pages 与小红书发布字段分开，但共用同一事实包。
3. 没有证据的项目不生成可发布卡片，而是进入待核对队列。
4. 保留 Schema 3.0 读取过渡层，直到历史报告和站点消费者完成迁移。

## 13. 验收指标

### 13.1 事实与安全门槛

- 仓库身份、URL、日期、排名、Star、Fork、增量口径、语言和许可证 **100%** 与冻结事实一致。
- 所有可发布能力都来自程序允许的受控主张；不支持主张数量为 **0**。
- 无证据时必须出现审核兜底；用泛话代替审核状态的数量为 **0**。
- 禁用夸张词、假亲测、伪性能结论和元数据冒充功能的数量为 **0**。
- README 文字不能触发工具、命令、文件读写、网络请求或输出 Schema 外字段。

### 13.2 可理解性门槛

每期发布前对至少 5 名未参与生成的读者做快速检查：

1. 只看单项目卡 5 秒后，能否回答“它替谁做什么”。
2. 能否说出至少一个具体结果。
3. 能否判断自己是否适用。

验收线：至少 4/5 读者正确完成前两项；如未达到，当期不发布并重写。

### 13.3 视觉门槛

- 手机缩略图能识别项目名和一句用途。
- 项目用途的视觉优先级高于 Star/Fork/Topics。
- 不存在裁切、重叠、破图、CJK 缺字、过小正文或假零值。
- 小红书版没有裸 URL、二维码或无权利证据的第三方视觉资产。

## 14. 三期人工 A/B 复盘

三期试验用于建立自身基线，不声称可推断平台算法。由于每期项目热度不同，结果只作方向性判断。

### 第 1 期：V2 基线

- 使用新的白话用途候选和单项目卡。
- 标题使用稳定的“榜单 + 数量 + 用途”结构。
- 记录人工可理解性测试和发布后 24 小时/7 天数据。

### 第 2 期：只测试封面/标题

- 保持卡片信息结构不变。
- 对比“榜单型标题”与“读者问题型标题”。
- 只在封面层改变一个变量，避免无法判断原因。

### 第 3 期：只测试收藏结构

- 使用已选定的标题/封面结构。
- 测试加入“怎么选/使用前需要什么”比较卡。
- 观察收藏、有效评论和下一期选题反馈是否改善。

### 每期记录字段

- 榜单、日期、项目数、标题版本、封面版本、卡片版本、发布时间。
- 当期可理解性测试结果、人工审核修改数、待核对项目数。
- 创作者中心实际可见的曝光、阅读/观看、收藏、点赞、评论、关注和搜索来源等指标。
- 24 小时与 7 天两个固定观测点。
- 读者真实提问、纠错、想要的下期主题和人工复盘结论。

如某个指标在当前创作者中心不可见，记录为“不可用”，不使用估算值代替。在建立自身基线前，不设置“收藏率必须提升某个百分比”类凭空目标。

## 15. 合规、版权与 AI 辅助标识

### 15.1 事实与体验

- 未运行、未安装或未复现的项目不写“我亲测”“我在用”“一分钟上手”。
- 如未来增加本地测试，必须保留环境、版本、命令、结果和日期，并将“README 声称”与“本地实测”分开。
- 不根据 Star 数推导安全性、可靠性、社区满意度或企业采用。

### 15.2 外链与发布行为

- 小红书图片和正文不放裸 URL、二维码、外部联系方式或诱导私信文字。
- 不自动登录、发布、点赞、收藏、评论或维护账号。
- 人工审核人员对平台当前界面和规则做最终判断。

### 15.3 视觉资产和标识

- 不复制用户参考帖的版式、配色、图形、文案或账号标识；只借鉴“它能做什么/核心亮点/适合谁”的信息架构。
- 仓库 LICENSE 默认仅覆盖代码，不自动覆盖 logo、商标、头像、截图或 README 素材。
- 项目 logo 只在仓库明确提供资产、来源可记录且权利状态经人工确认后使用。
- 无可确认资产时，使用由仓库名稳定生成的原创抽象图形，不直接抓取第三方图片。
- 项目采用 MIT 不代表“商用无忧”；商标、专利、依赖和第三方素材仍需独立核对。

### 15.4 AI 辅助标识

每期发布物的尾卡或正文末尾保留类似文字：

> AI 辅助整理，人工审核；排名与数字来自 GitHub 和本地快照。

最终发布时，人工审核人员应按小红书当前客户端的 AI 辅助内容入口主动标明。

## 16. 已知限制

- 当前 V2 仍主要依赖结构化仓库事实、精确 Topics 和受控短语规则，尚不能覆盖每个仓库的完整功能。
- 评分高或 Star 增长不代表项目适合所有人，也不代表安全或稳定。
- 三期 A/B 只能建立本账号的初步内容基线，不能推断平台黑箱算法。
- 小红书规则、字段限制和创作者中心指标可能变化，每次真实发布前需人工确认。
