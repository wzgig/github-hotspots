# 小红书内容 V2 改进计划

> 状态：Prompt 4.1 / Schema 4.0 与 V4 项目卡已实现，本文作为当前执行规范
> 适用范围：“综合主榜 + AI 专题榜”的小红书文案、轮播卡片与人工发布审核
> 当前兼容性：Codex 输出使用 Schema 4.0；`highlights[3]` 继续作为旧报告兼容字段，正式项目卡优先使用 `capabilities[1..5]`、核心亮点、受众、前置条件和限制

## 1. 决策摘要

当前内容的主要矛盾不是“文案不够热闹”，而是“数据卡没有完成用途解释”。Star、Fork、语言和 Topics 可以证明仓库状态，却不能自动回答读者最关心的三个问题：

1. 它替谁工作？
2. 它具体做什么？
3. 使用后得到什么结果？

V2 将以“替谁做什么，得到什么”作为所有项目解释的最小闭环，并采用以下已经落地的路线：

- 证据化编辑：Prompt 4.1 / Schema 4.0 可以在同仓库清洗后的 README/metadata 证据范围内改写白话内容，不再把 Codex 限定为逐字符选择候选。
- 内容分面：Pages 详情卡和小红书轮播卡使用不同信息密度和发布规则，不再让一张“工程仪表盘”同时承担两种任务。
- 证据优先：每个自然语言字段都绑定合法 `evidence_ids`；不能确认时使用 `needs_review`，不用泛话伪装成能力。
- 人工决策：自动化仅到“生成可审核文案和配图”；人工负责事实、权利、AI 辅助标识和最终发布。
- 原创视觉：V4 项目卡采用与网站一致的 Signal Broadsheet 品牌，以硬边报头、信号数据条和五项能力轨道组织内容；Owner 头像经过域名、体积、像素、格式和路径检查，失败时使用确定性身份块。

### 2.2 不得写成“确定机制”的说法

以下说法缺少可靠公开证据，本项目的文档、Prompt、站点和发布文案不得将其当作事实：

- 固定的“冷启动流量池”规模、轮次和晋级阈值。
- 所谓 CES 等互动权重公式，以及点赞、收藏、评论的固定倍数。
- “发布后前两小时决定全部流量”或某个必胜发布时间。
- 某个固定字数、标签数量或发布频次必然获得推荐。
- 使用“神器”“必装”“爆火”等词本身能带来更高流量。

本项目可以优化可观察的内容质量，例如首屏可理解性、读完后的任务理解、收藏价值和评论问题质量；不得声称这些信号在平台排序中的具体权重。

## 3. 受众与阅读任务

### 3.1 核心受众

| 受众 | 首要问题 | 内容必须提供 |
| --- | --- | --- |
| 非技术的 AI/效率工具用户 | 它能替我省哪件事？ | 具体任务、结果形式、使用前提 |
| 学生、初学者、知识工作者 | 我看不懂 CLI、MCP、Agent 等词，它和我有什么关系？ | 术语白话解释、典型场景、是否需要编程 |
| 开发者和技术评估者 | 它的功能边界、依赖和许可条件是什么？ | 语言、部署/依赖、许可证、已知限制、证据来源 |
| 收藏型读者和选型者 | 这几个项目有什么区别，哪个适合我？ | 用途/受众/门槛/限制比较，可搜索的 `owner/repo` 标识 |

### 3.2 阅读任务顺序

每组内容应让读者依次完成：

1. **3 秒识别**：知道这是哪个榜单、本期有几个项目、自己能得到什么。
2. **10 秒理解**：能用自己的话复述单个项目“替谁做什么，得到什么”。
3. **30 秒判断**：知道自己是否适用，以及使用前需要什么。
4. **滑完选择**：在同期项目中做初步比较，决定收藏、进一步搜索或提问。

## 4. 内容模型：从元数据卡到任务解释卡

### 4.1 最小解释闭环

单项目必须形成以下链条：

```text
谁在什么场景遇到问题
  -> 项目执行什么动作
  -> 用户得到什么结果
  -> 使用前需要什么
  -> 哪些是确定事实，哪些仍待核对
```

`one_line` 不要强制使用同一句式，但必须同时包含“使用者/场景、动作、结果”。示例句式：

- 给 `[specific audience]` 用来 `[action]`，完成后得到 `[result]`。
- 当 `[scenario]` 时，它会 `[action]`，帮助 `[audience]` 得到 `[result]`。
- 把 `[input]` 变成 `[output]`，适合需要 `[task]` 的 `[audience]`。

不合格句式：

- `X 是一个面向工程实践方向的开源项目`
- `X 是一个值得关注的 AI 工具`
- `X 以 Python 为主要语言，本周新增 500 Star`

第三句只提供元数据，不是用途解释。

### 4.2 当前 Schema 4.0 字段语义

| 字段 | 当前含义 | 不得替代为 |
| --- | --- | --- |
| `one_line` | 回答“替谁做什么，得到什么”的白话摘要 | Star 变化、主要语言、Topics 列表或泛化类别 |
| `highlights[3]` | 为历史报告保留的三条互不重复核心能力 | 重复 `one_line`、重复数字、简单罗列 Topics |
| `capabilities[1..5]` | 海报“它能做什么”区域的具体任务；证据充分时优先写满 5 条 | 把语言、Star、Fork、排名或许可证当成功能 |
| `core_title` / `core_summary` | 最有辨识度的工作流或能力，以及它如何运作、为什么有用 | 空洞口号、未经证实的性能或“全自动”夸张表述 |
| `audience` | 具体任务场景中的人，优先描述“要做什么” | 过宽的“开发者、技术团队、开源爱好者” |
| `prerequisites` / `limitations` | README 明确写出的前置条件、人工确认点或能力边界 | 模型自行推测的门槛或限制 |
| `license_label` / `license_restrictions` | 明确许可证标识与限制原文 | 把 `NOASSERTION` 猜成 MIT，或推导“商用无忧” |
| `readme_sha` / `content_status` | 标识本次 README 版本和 `readme_enriched`、`metadata_only`、`needs_review` 状态 | 用模糊措辞掩盖证据缺失 |
| `evidence_ids` | 逐字段列出本仓库允许引用的证据 ID | 笼统写“README”“网页”或跨仓库引用 |

如果受控信息不足以确认用途，安全兜底必须明示说明并使用 `needs_review`：

- `one_line`：“公开信息不足，暂不能准确说明具体用途”。
- `highlights`：分别标记需核对 README 定位、输入/输出和适用场景。
- `audience`：“需人工确认后再发布”。

这比“开源软件与工程实践项目”更诚实，也不会让读者将类别名误解为项目能力。README 缺失时，Codex 只能逐字段采用单一受控候选，不能借 metadata 自由补写。

## 5. 小红书轮播结构

综合主榜和 AI 专题榜继续分开生成文案与图片，同一仓库可在两榜同时出现。

### 5.1 日榜

建议 6–7 张：

1. **封面**：榜单、日期/期号、“3 个项目分别解决什么问题”。
2. **先看结论**：三个项目各一句用途 + 适用人群。
3. **项目 1**：单项目详解卡。
4. **项目 2**：单项目详解卡。
5. **项目 3**：单项目详解卡。
6. **怎么选**：用途 / 适合谁 / 使用前需要什么。
7. **方法与互动**：数据窗口、人工审核、AI 辅助标识、下期问题。

如果暂时不能增加结论卡，最少保留“封面 + 3 张项目卡 + 方法/互动卡”。

### 5.2 周榜

建议 10–11 张：

1. 封面。
2. 先看结论。
3. 至 9. 七张独立项目卡。
10. 横向比较/选型卡。
11. 方法、限制、AI 辅助标识和互动卡。

发布前由人工按小红书当前客户端允许的图片数量和字段限制复核，不在程序中把未持续核验的平台上限当成永久事实。

## 6. 单项目卡规范

卡片的信息顺序必须是“用途优先，数字次之”：

1. **顶部识别**：`#排名`、`owner/repo`、榜单标识。
2. **项目身份卡**：GitHub Owner 头像、项目名、说人话摘要和许可证；许可证无证据时显示“许可证未标注”，不猜 MIT。
3. **四项统计**：总 Star、语言、周期增长、Fork；增长标签必须忠实区分 `snapshot`、`trending` 和 `estimate`。
4. **它能完成什么**：1 至 5 条具体能力，证据充分时优先写满 5 条；每条有信息差，不重复摘要。
5. **核心亮点**：`core_title` 加 `core_summary`，解释最有辨识度的工作流如何运作、为什么有用。
6. **适合谁**：用任务场景描述，不只给宽泛职业名。
7. **文字审核稿补充**：有证据的依赖、API Key、人工确认点或已知限制写入正文，不为塞进海报而缩成难读小字；暂无证据时不编造。
8. **系列页脚**：系列标识和日期；不放裸 URL、内部管线术语或重复的数据来源声明。

### 6.1 视觉约束

- 使用 `1200×1600` 的 3:4 竖版，固定复现参考图的信息比例与层级：深色系列页眉、白色项目身份卡、四个统计卡、左侧能力大卡、右侧核心亮点/适合谁卡和底部日期。不得复制参考帖原图、账号标识或受保护文案。
- 手机缩略图中必须看清项目名和一句用途；次要数字不得抢过用途。
- 单项目卡优先一屏完成解释；不为填满版面增加小字。
- 封面不展示 `00 Trending`、`00 估算`等零值或内部数据状态。
- 不把装饰性英文当作主标题；技术专名可保留，首次出现时要解释。
- 项目名左侧优先显示安全缓存的 GitHub Owner 头像；头像下载或校验失败时显示确定性占位图，不能热链远程图片。

## 7. 标题与首屏

### 7.1 标题公式

标题使用“范围 + 数量/榜单 + 读者收益”，优先包含自然搜索词：

- `GitHub AI 周榜｜7 个项目到底能做什么`
- `别只看 Star｜本周 GitHub 热门项目讲人话`
- `3 个 GitHub 热门项目：用途、门槛、适合谁`
- `开发者周报｜7 个开源项目怎么选`

内部设计目标是“一眼读完”，发布前再按客户端当前字符限制人工调整。

禁止：

- 全网爆火、所有人都在用、必装、封神、神器、宝藏、零门槛。
- 没有定义数据窗口时的“一周暴涨”。
- 用 Star 数直接证明“最好用”“生产级”或“商用无忧”。

### 7.2 封面信息

封面只保留：

- 榜单名称和日期/期号。
- 一句明确收益，例如“3 个项目分别解决什么问题”。
- 一句内容承诺，例如“每个项目讲清：用途 · 适合谁 · 使用条件”。
- 最多 2–3 个本期真实任务词，例如“会议整理”“终端自动化”“代码测试”。

排名方法、快照质量和详细数据放在尾卡或正文末尾，不占用封面的首要注意力。

## 8. 正文规范

正文使用以下顺序：

1. **读者问题**：前两行写“看到 GitHub 榜单但不知道项目能做什么”等真实困惑。
2. **本期承诺**：说明会讲清用途、受众和使用条件，不只报 Star。
3. **项目摘要**：每个项目一行，格式为“项目名 - 具体任务 - 适合谁”。
4. **比较/推荐**：只根据已证实的任务和条件做差异化总结，不用主观“最强”排序。
5. **方法和边界**：标明数据窗口、快照/估算口径、AI 辅助和人工审核。
6. **互动问题**：提出一个可回答、可帮助下期选题的问题。
7. **标签**：使用少量精确主题，不堆叠无关热词。

小红书版不放裸 URL、二维码或外部联系方式。为便于读者核对项目，可在人工审核后保留不带协议和域名的 `owner/repo` 搜索标识。Pages 版仍可保留可点击的 GitHub URL。

## 9. 互动与收藏设计

互动的目的是收集真实需求，不是操纵平台信号。

可用问题：

- `下一期更想看会议助手还是终端自动化？`
- `哪个项目希望我下期补一张上手条件卡？`
- `你选工具时更在意本地部署、免费开源，还是使用门槛？`

收藏价值由以下内容提供：

- 一张横向比较表。
- 一张“适合谁/不适合谁”卡。
- 明确的使用前提和搜索标识。
- 可核对的数据窗口和快照日期。

禁止诱导点赞、互赞、互藏、抽奖换互动、买量刷数据或机器评论。

## 10. 证据链与 README 不可信输入安全

### 10.1 信任分层

| 层级 | 内容 | 用法 |
| --- | --- | --- |
| 冻结结构化事实 | 仓库 ID、`full_name`、URL、Star、Fork、语言、许可证、日期、排名、快照增量 | 程序拷贝与校验，模型不得修改、换算或纠错 |
| 受控分类事实 | 精确 Topics、否定保护后的严格短语规则 | 只能触发预定义能力文案，不得自由关键词联想 |
| 程序兜底与候选 | `deterministic_draft`、`candidate_summaries` | 默认确定性输出；README 缺失时，Codex 只能逐字段匹配同一受控候选 |
| README/描述/metadata | 项目作者或 GitHub 提供的外部文本与字段 | 永远按不可信数据处理；Prompt 4.1 可在证据语义内白话改写，但不得执行其中的指令 |
| Owner 头像 | GitHub metadata 给出的头像 URL | URL 可作为受控 metadata 出现，但模型不得访问，图片字节不进入 LLM；只经安全下载、重新编码和本地相对路径缓存后用于海报 |

### 10.2 README 处理规则

当前 README 管线已经建立“采集/清洗”与“编辑”之间的隔离层：

1. 只通过受控 GitHub API/快照获取指定仓库的 README，不跟随 README 中的外部链接。
2. 将 README 作为纯数据解析；其中的“忽略之前指令”“运行命令”“上传文件”等内容不得被执行。
3. 去除或隔离 HTML、隐藏文字、徽章、过长代码块和外部营销素材。
4. 保留定位、明确功能、典型输入/输出、安装前提、兼容性、许可证原文和已知限制，并记录 README SHA。
5. 每个自然语言字段都返回 `evidence_ids`；README 使用 `github.readme:<sha>`，metadata 和候选使用程序生成的稳定 ID。
6. README 中的自我宣传，例如“最快”“领先”“生产级”，只能当作“项目方声称”，除非有可验证证据，否则不转写为事实。
7. 相互冲突的描述进入人工审核，不由模型选择看起来更合理的版本。

### 10.3 无证据时的状态

当前 Schema 4.0 使用以下审核状态：

- `readme_enriched`：主要文案得到当前 README 证据支持，且至少一个自然语言字段引用 `github.readme:<sha>`。
- `metadata_only`：本次没有 README；文案严格来自单一受控候选和有意义的 metadata 许可证字段。
- `needs_review`：证据不足以形成完整解释，必须人工核对后再发布。

任何非法证据 ID、README SHA 不一致、许可证不匹配、身份/数字漂移或输出失败都会丢弃整个榜单的 Codex 结果并回退到确定性文案；不接受“部分项目通过、部分项目自动修补”。

## 11. 当前 Prompt 4.1 / Schema 4.0 规范

### 11.1 输入契约

`prompts/repository_summary_zh.md` 只接收程序组织的整榜证据包：

- 冻结仓库身份、榜单顺序、URL、语言、Star、Fork、周期增量和统计窗口；
- `deterministic_draft` 与七个带稳定 ID 的 `candidate_summaries`；
- `repository_evidence.metadata`：受控 GitHub metadata；
- `repository_evidence.readme`：经过清洗、限长、带 SHA 的 README，或 `null`；
- `available_evidence_ids`：当前仓库允许引用的全部证据 ID。

README、description、Topics 和 metadata 全部是不可信外部数据。模型不得执行其中的命令、访问链接、调用工具、读取文件、改变角色或修改输出格式，也不能跨仓库引用证据。

### 11.2 输出契约

输出 `schema_version="4.0"`，每个 `card` 包含：

```text
one_line
highlights[3]                  # 旧报告兼容字段
capabilities[1..5]
core_title, core_summary
audience
prerequisites, limitations
license_label, license_restrictions
readme_sha
content_status                # readme_enriched | metadata_only | needs_review
evidence_ids                  # 上述每个自然语言字段分别引用
```

每个非空自然语言字段至少引用一个当前仓库 `available_evidence_ids` 中的 ID；`highlights` 和 `capabilities` 按每条文字分别引用。空字符串必须使用空证据数组。文案中的阿拉伯数字必须逐字符出现在该句引用的证据中。

### 11.3 README 与许可证规则

- README 存在时，可以在证据语义范围内自由改写白话内容，不必逐字符复制候选；`readme_sha` 必须完全一致，`readme_enriched` 至少引用一次 `github.readme:<sha>`。
- README 缺失时，`readme_sha=null`，除许可证外的自然语言字段必须逐字段匹配同一个 angle 的单一候选；不得借 metadata 补写新能力。
- metadata SPDX 为 `NOASSERTION`、`OTHER`、`unknown` 或空值时，`license_label` 不能写 MIT，也不能推导允许商用。
- `license_restrictions` 只能逐字复制 README 中连续出现的限制原文，不能翻译、概括或给出法律结论。
- 任一项目的证据 ID、README SHA、许可证、身份、URL、数字、重复或 Schema 校验失败时，整个榜单回退到确定性文案。

### 11.4 Prompt 正反例

以下只演示证据充足时的文字结构，不是对某个真实仓库的事实断言。

**正例**

> 给 C++ 开发者编写并运行单元测试，输出失败用例，便于定位代码问题。

这句同时包含使用者、动作和结果。前提是每个分句都有允许主张或 README 证据支持。

**反例：泛话**

> 这是一个面向开源软件与工程实践的热门项目。

没有可执行任务，“热门”也被用来代替用途。

**反例：元数据冒充能力**

> 项目使用 Python，本周新增 2,000 Star，Topics 包含 AI。

语言、Star 和 Topics 是事实栏，不能说明项目做什么。

**反例：无证据亲测**

> 我亲测安装简单、运行稳定，值得所有人使用。

如果没有可复现本地测试记录，整句都不得发布。

**正例：证据不足**

> 公开信息不足，暂不能准确说明具体用途；需人工核对 README 后再发布。

明示不确定比创造一个听起来合理的功能更安全。

## 12. 已实施能力与后续路线

### 12.1 已实施：Prompt 4.1 / Schema 4.0 证据化出版

1. 从 GitHub API 获取受控 metadata、README SHA/正文和 Owner 头像 URL。
2. 清洗 README，限制体积、长度和代码块，并加入不可信数据警告。
3. 允许 Codex 在证据语义范围内写白话定位、最多 5 条能力、核心亮点、受众、前置条件和限制。
4. 冻结身份、URL、数字、排名和许可证事实；为自然语言字段逐项校验证据 ID。
5. README 缺失时限制到单一受控候选，任一批次校验失败时整榜确定性回退。
6. 报告读取层继续兼容历史 Schema，但当前 Codex 输出契约固定为 4.0。

### 12.2 已实施：Signal Broadsheet V4 原创信息架构

1. 米纸、黑墨、信号橙与酸绿色继承项目页面品牌；综合榜为浅色热力格，AI 榜为深色雷达。
2. 硬边报头同时显示 `ALL/AI`、`24H/7D` 和 `Dxxx/Wxxx`，缩略图不只靠颜色区分。
3. 安全缓存的 Owner 头像、项目名和白话定位构成首屏焦点；增长、Star、Fork、语言与许可证压缩为一条 signal bar。
4. Git 式 Signal Rail 按 01—05 串联五项能力；核心亮点使用跨栏信号带，受众放在底部，不再使用旧版左右分栏。
5. 底部只保留品牌、仓库搜索词和日期，不展示裸 URL、内部审核状态或重复来源声明。

### 12.3 下一阶段

1. 在人工审核界面显示每个字段与证据原文的对照，而不是只保存 ID。
2. 建立不少于 30 个仓库的固定评估集，量化用途可理解性、证据覆盖、许可证准确性和整榜回退率。
3. 将 Pages 详情字段与小红书发布字段进一步分面，但继续共用同一冻结事实与证据包。
4. 完善文件化 `draft`、`approved`、`rejected` 审核状态；自动发布仍不在当前范围。

## 13. 验收指标

### 13.1 事实与安全门槛

- 仓库身份、URL、日期、排名、Star、Fork、增量口径、语言和许可证 **100%** 与冻结事实一致。
- 所有可发布能力都引用当前仓库合法 `evidence_ids`；不支持主张数量为 **0**。
- README 缺失分支必须逐字段匹配同一受控候选；越界改写数量为 **0**。
- 许可证缺失或模糊时不得猜测 MIT 或商用权利；许可证误报数量为 **0**。
- 无证据时必须出现 `needs_review` 审核边界；用泛话代替审核状态的数量为 **0**。
- 禁用夸张词、假亲测、伪性能结论和元数据冒充功能的数量为 **0**。
- README 文字不能触发工具、命令、文件读写、网络请求或输出 Schema 外字段。

### 13.2 可理解性门槛

每期发布前对至少 5 名未参与生成的读者做快速检查：

1. 只看单项目卡 5 秒后，能否回答“它替谁做什么”。
2. 能否说出至少一个具体结果。
3. 能否判断自己是否适用。

验收线：至少 4/5 读者正确完成前两项；如未达到，当期不发布并重写。

### 13.3 视觉门槛

- 手机缩略图能识别项目名和一句用途。
- 项目用途的视觉优先级高于 Star/Fork/Topics。
- 不存在裁切、重叠、破图、CJK 缺字、过小正文或假零值。
- 小红书版没有裸 URL、二维码或无权利证据的第三方视觉资产。
- Owner 头像只从本地安全缓存读取；远程热链、路径越界和解码失败均使用占位图。

## 14. 三期人工 A/B 复盘

三期试验用于建立自身基线，不声称可推断平台算法。由于每期项目热度不同，结果只作方向性判断。

### 第 1 期：Signal Broadsheet 基线

- 使用新的白话用途候选和单项目卡。
- 标题使用稳定的“榜单 + 数量 + 用途”结构。
- 记录人工可理解性测试和发布后 24 小时/7 天数据。

### 第 2 期：只测试封面/标题

- 保持卡片信息结构不变。
- 对比“榜单型标题”与“读者问题型标题”。
- 只在封面层改变一个变量，避免无法判断原因。

### 第 3 期：只测试收藏结构

- 使用已选定的标题/封面结构。
- 测试加入“怎么选/使用前需要什么”比较卡。
- 观察收藏、有效评论和下一期选题反馈是否改善。

### 每期记录字段

- 榜单、日期、项目数、标题版本、封面版本、卡片版本、发布时间。
- 当期可理解性测试结果、人工审核修改数、待核对项目数。
- 创作者中心实际可见的曝光、阅读/观看、收藏、点赞、评论、关注和搜索来源等指标。
- 24 小时与 7 天两个固定观测点。
- 读者真实提问、纠错、想要的下期主题和人工复盘结论。

如某个指标在当前创作者中心不可见，记录为“不可用”，不使用估算值代替。在建立自身基线前，不设置“收藏率必须提升某个百分比”类凭空目标。

## 15. 合规、版权与 AI 辅助标识

### 15.1 事实与体验

- 未运行、未安装或未复现的项目不写“我亲测”“我在用”“一分钟上手”。
- 如未来增加本地测试，必须保留环境、版本、命令、结果和日期，并将“README 声称”与“本地实测”分开。
- 不根据 Star 数推导安全性、可靠性、社区满意度或企业采用。

### 15.2 外链与发布行为

- 小红书图片和正文不放裸 URL、二维码、外部联系方式或诱导私信文字。
- 不自动登录、发布、点赞、收藏、评论或维护账号。
- 人工审核人员对平台当前界面和规则做最终判断。

### 15.3 视觉资产和标识

- V4 使用项目自有的 Signal Broadsheet 品牌；禁止恢复旧版深绿头部、圆形排名章、四格统计和左右深浅分栏，也不复制任何参考帖的原图、账号标识、受保护文案或视觉组合。
- 仓库 LICENSE 默认仅覆盖代码，不自动覆盖 logo、商标、头像、截图或 README 素材。
- 项目 logo 只在仓库明确提供资产、来源可记录且权利状态经人工确认后使用。
- 项目名称左侧允许使用 GitHub Owner 公开头像，但必须经受控域名、下载体积、像素、Content-Type、重定向、路径和重新编码门禁，并在人工发布前复核；失败时使用由仓库名稳定生成的原创占位图。
- 项目采用 MIT 不代表“商用无忧”；商标、专利、依赖和第三方素材仍需独立核对。

### 15.4 AI 辅助标识

每期发布物的尾卡或正文末尾可保留简短说明：

> AI 辅助整理｜人工发布

不在每张图片或项目段落重复“数据来自 GitHub 公开信息与本地快照”；数据窗口、增量类型和方法说明仍在报告与 Pages 中可核验。

人工复核仍是发布前的内部硬门禁；最终发布时，还应按小红书当前客户端的 AI 辅助内容入口主动标明。

## 16. 已知限制

- README 可能缺失、过长、表述含糊或只描述部分功能；此时文案会降级为受控候选或 `needs_review`，不能保证覆盖仓库全部能力。
- GitHub API 的许可证 `NOASSERTION` 不能说明实际许可状态；仍需人工检查仓库许可证文件和第三方依赖。
- Owner 头像属于公开外部视觉资产，安全缓存只解决技术风险，不替代发布前的权利与平台规则复核。
- 评分高或 Star 增长不代表项目适合所有人，也不代表安全或稳定。
- 三期 A/B 只能建立本账号的初步内容基线，不能推断平台黑箱算法。
- 小红书规则、字段限制和创作者中心指标可能变化，每次真实发布前需人工确认。
